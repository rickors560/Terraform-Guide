# Contributing

Thank you for your interest in contributing to the Terraform AWS Masterclass. This document provides guidelines and conventions to keep the codebase consistent and the review process efficient.

---

## Getting Started

1. Fork the repository and clone your fork locally.
2. Install all prerequisites listed in [PREREQUISITES.md](PREREQUISITES.md).
3. Install pre-commit hooks:
   ```bash
   pre-commit install
   ```
4. Create a feature branch from `main` (see branch naming below).

---

## Branch Naming

Use the following pattern:

```
{type}/{short-description}
```

| Type | Purpose | Example |
|---|---|---|
| `feature/` | New functionality | `feature/add-rds-module` |
| `fix/` | Bug fix | `fix/sg-ingress-rule` |
| `docs/` | Documentation only | `docs/update-glossary` |
| `refactor/` | Code restructuring | `refactor/module-outputs` |
| `chore/` | Tooling, CI, dependencies | `chore/update-tflint` |
| `phase/` | Phase implementation | `phase/3-eks-cluster` |

---

## Commit Messages

Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
{type}({scope}): {description}

[optional body]

[optional footer]
```

### Types

| Type | Description |
|---|---|
| `feat` | New feature or resource |
| `fix` | Bug fix |
| `docs` | Documentation changes |
| `style` | Formatting, whitespace (no logic change) |
| `refactor` | Code restructuring (no feature or fix) |
| `test` | Adding or updating tests |
| `chore` | Tooling, CI, build system |

### Examples

```
feat(networking): add VPC module with three-tier subnet design

- Public, private, and data subnets across 3 AZs
- NAT Gateway per AZ for high availability
- VPC Flow Logs to CloudWatch

Closes #12
```

```
fix(eks): correct node group subnet assignment

Node groups were using public subnets instead of private.
This caused worker nodes to receive public IPs.
```

```
docs(glossary): add ElastiCache and Fargate terms
```

## Code Standards

### Terraform

- **Format:** Run `terraform fmt -recursive` before committing. The pre-commit hook enforces this.
- **Naming:** Use snake_case for all resource names, variable names, and output names.
- **Resource naming convention:** `{project}-{environment}-{component}-{qualifier}` using the `local.name_prefix` pattern:
  ```hcl
  locals {
    name_prefix = "${var.project_name}-${var.environment}"
  }
  ```
- **Variables:** Every variable must have a `description` and a `type`. Use `validation` blocks for constraints.
- **Outputs:** Every output must have a `description`.
- **Tags:** All resources must include the standard tag set:
  ```hcl
  tags = merge(var.tags, {
    Name        = "${local.name_prefix}-component"
    Component   = "component-name"
    ManagedBy   = "terraform"
  })
  ```
- **Modules:** Keep modules focused on a single concern. Use consistent file structure:
  ```
  modules/{name}/
  ├── main.tf          # Resources
  ├── variables.tf     # Inputs
  ├── outputs.tf       # Outputs
  ├── versions.tf      # Provider requirements
  ├── locals.tf        # Computed values (if needed)
  └── README.md        # Auto-generated by terraform-docs
  ```
- **State:** Never store secrets in Terraform state when avoidable. Use `sensitive = true` on sensitive outputs.

### File Organization

Each `.tf` file has a clear purpose:

| File | Contents |
|---|---|
| `main.tf` | Primary resources |
| `variables.tf` | Input variable declarations |
| `outputs.tf` | Output declarations |
| `versions.tf` | `terraform` and `required_providers` blocks |
| `locals.tf` | Local values and computed expressions |
| `data.tf` | Data sources |
| `providers.tf` | Provider configuration (root modules only) |

### Documentation

- Update `README.md` files when adding or changing modules.
- Run `terraform-docs markdown table . > README.md` to regenerate module docs.
- Keep the glossary alphabetized when adding new terms.

## Pull Request Process

### Before Opening a PR

1. **Rebase on latest `main`:**
   ```bash
   git fetch origin
   git rebase origin/main
   ```
2. **Run all checks locally:**
   ```bash
   make fmt
   make validate
   make lint
   ```
3. **Ensure pre-commit hooks pass:**
   ```bash
   pre-commit run --all-files
   ```

### PR Requirements

- **Title:** Use the same format as commit messages (`type(scope): description`).
- **Description:** Include:
  - Summary of changes
  - Related phase number (if applicable)
  - Testing performed (`terraform plan` output or screenshots)
  - Checklist of completed items
- **Size:** Keep PRs focused. One module or one component per PR. Large PRs are harder to review and more likely to require changes.

### PR Template

```markdown
## Summary

Brief description of what this PR does and why.

## Changes

- Added/Updated/Removed X
- Added/Updated/Removed Y

## Phase

Phase N — [phase name]

## Testing

- [ ] `terraform fmt` passes
- [ ] `terraform validate` passes
- [ ] `tflint` passes
- [ ] `checkov` passes
- [ ] `terraform plan` runs successfully
- [ ] `terraform apply` tested in dev environment (if applicable)

## Notes

Any additional context or considerations for reviewers.
```

### Review Process

1. At least one approval is required before merging.
2. All CI checks must pass (format, validate, lint, security scan).
3. Reviewers will check for:
   - Correct naming conventions and tagging
   - Security best practices (no public access, encryption enabled)
   - Module interface clarity (variables, outputs, descriptions)
   - Documentation completeness
4. Use "Squash and merge" for single-purpose PRs. Use "Merge commit" for multi-commit PRs where history is valuable.

---

## Reporting Issues

- Use GitHub Issues for bugs, feature requests, and questions.
- For security vulnerabilities, see [SECURITY.md](SECURITY.md).
- Label issues with the appropriate phase (e.g., `phase:3`, `phase:7`).

---

## Code of Conduct

Be respectful, constructive, and inclusive. Focus on the code, not the person. We are all here to learn.
